# ROADMAP

Goal: define and deliver the `0.6.5-SNAPSHOT` scope.

## Priority 1

Objective: adopt release-please — automate release PR creation, CHANGELOG generation, version bumps, tagging, and GitHub Releases from Conventional Commits. Remove all custom release scripting and label-based release note infrastructure.

### What changes

The current release pipeline relies on custom scripts (`ai-release.sh prepare/finalize`, `compose-release-notes.sh`), manual version bumps (`mvn versions:set`), PR-body extraction of release notes, label-based categorization, and a `central.skipPublishing` pom.xml property. All of this is replaced by release-please:

- Conventional Commits on `main` are the single source of truth.
- release-please automatically creates and maintains a Release PR (version bump in all pom.xml + CHANGELOG.md update).
- Merging the Release PR triggers: GitHub Release creation, tag creation, Maven Central publish.
- No publish happens when feature PRs are merged to `main` — only when the Release PR is merged.
- `--skip-central` / `central.skipPublishing` become unnecessary because the publish timing is controlled entirely by when the Release PR is merged.

### Tasks

#### T-001 — Add release-please configuration

Create the required configuration files:

- `release-please-config.json`:
  - `release-type`: `maven`
  - `changelog-sections`: map Conventional Commit types to sections (Features, Bug Fixes, Performance, Documentation, Breaking Changes)
  - Exclude internal types from CHANGELOG: `chore`, `test`, `ci`, `build`, `refactor`, `revert`
- `.release-please-manifest.json`:
  - Initialize with `{ ".": "0.6.4" }` (the last released version)

#### T-002 — Add release-please GitHub Action

Add a new workflow (or integrate into `ci.yml`):

- Use `googleapis/release-please-action@v4` on pushes to `main`.
- Outputs: `release_created`, `tag_name`, `version`, `pr` (PR number).
- release-please manages its own branch (`release-please--branches--main`) and PR lifecycle.

#### T-003 — Restructure CI pipeline

Restructure `.github/workflows/ci.yml`:

- Remove the `tag-version` job entirely — release-please creates tags.
- Replace the `release` job with a `publish` job that:
  - Only runs when release-please output `release_created == 'true'`.
  - Publishes to Maven Central: `mvn -B -Prelease -DskipTests deploy`.
  - No custom release note generation, no tag creation, no draft-release handling.
- The `build` and `sonar` jobs remain unchanged.

#### T-004 — Remove custom release scripts

Delete scripts that release-please replaces:

- `scripts/ai-release.sh` — prepare/finalize flow is obsolete.
- `scripts/compose-release-notes.sh` — CHANGELOG + release notes are generated by release-please.

#### T-005 — Simplify `scripts/ai-pr.sh sync`

Adapt for feature PRs only (release PRs are managed by release-please):

- Remove the "Release Notes" section generation from PR body — release notes live in CHANGELOG.md now.
- Remove the "Type of Change" checkbox section.
- Remove the "Scope by Commit Type" section.
- Keep: Summary, Breaking Changes (from `!` commits), Included Commits, Test Plan.

#### T-006 — Simplify PR template

Update `.github/PULL_REQUEST_TEMPLATE.md`:

- Remove the "Type of Change" checklist.
- Remove the "Release Notes" section and its HTML comment about label-based workflow.
- Keep: Summary, Breaking Changes, Test Plan.

#### T-007 — Remove label-based release configs

- Delete `.github/release-drafter.yml` — replaced by release-please.
- Delete `.github/release.yml` — label-based changelog categories are obsolete.

#### T-008 — Remove `central.skipPublishing` mechanism

- Remove the `<central.skipPublishing>` property from root `pom.xml`.
- Remove the `<skipPublishing>` usage from the central-publishing-plugin configuration.
- Remove `set_central_skip_publishing()` references (already gone with T-004).
- Publish control is now purely timing-based: merge the Release PR when ready to publish.

#### T-009 — Update `CLAUDE.md`

Update project rules to reflect the new release model:

- Replace "Release Rules" section: release-please manages version bumps, tags, CHANGELOG, and GitHub Releases.
- Replace "Release Safety" section: no more manual tag creation, no finalize step.
- Remove two-phase prepare/finalize workflow references.
- Remove `mvn versions:set` instructions for releases.
- Remove `compose-release-notes.sh` and `ai-release.sh` references.
- Update "CI Pipeline" section to describe the new flow: `build` → `sonar` → `release-please` → `publish`.
- Update "PR policy" to note that release PRs are managed by release-please, feature PRs use `ai-pr.sh sync`.

#### T-010 — Bootstrap CHANGELOG.md

- Create an initial `CHANGELOG.md` with a note that prior releases are available in GitHub Releases.
- release-please will prepend new entries automatically from this point forward.

### Out of scope

- Monorepo / per-module releases — all modules release together as one version.
- Automatic SNAPSHOT bump after release — release-please handles this via its own PR cycle, or it can be omitted if SNAPSHOT versions are not needed between releases.
- Migration of historical release notes into CHANGELOG.md.
